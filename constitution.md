## 第3条：更新範囲の絶対上限

flow_weightの値域：[0.0, 1.0]
実装上のクランプ上限：0.9（推奨上限ではなく設計上限）
実装上のクランプ下限：0.01
1回あたりの更新幅の上限：±0.3
RCが設定する上限を超える更新は自動的に却下される。

### AAS更新則の標準形式（明文化）

```
w_new = w_old + α × reward + ε
ε ~ N(0, σ²)

パラメータ：
  α（学習率）   ：RC設定ファイルで管理
  σ（ゆらぎ）   ：RC設定ファイルで管理（SIGMA=0.05, SIGMA_MAX=0.1）
  reward        ：成功時=α×(1.0-w_old)、失敗時=-(w_old×(1-DECREASE_RATE))
```

この更新則はHebbian学習に近い形式であり
キメラ状態が自発出現するリスクがある。
（参照：Anand et al. 2026, arXiv:2603.10668）

### キメラ状態への対処

キメラ状態とは：
  均一なネットワークで一部のノードが同期・
  一部が非同期になる状態が自発的に出現する現象。
  「正常に見えるが実は一部のAASが暴走している」
  状態が生まれる可能性がある。

検出方法：
  flow_weightの分布のbimodality係数（二峰性）を監視する。
  二峰性が検出された場合 → 人間に即時通知する。
  全体の分散だけでは検出できないため
  第4条の監視対象3をbimodality係数に拡張する。

更新則の変更権限：
  AAS更新則の形式変更は人間のoffline操作のみ許可する。
  RCは更新則の形式を変更できない。

## 第4条：RCの監視対象リスト（固定）

RCが監視する指標は以下に限定される：

  1. 各腕のflow_weightの現在値
  2. 各腕の直近N回の正解率
  3. flow_weightの分布（キメラ状態検出）
     - 全体の分散（多様性の指標）
     - bimodality係数（二峰性の検出）
       → 二峰性が検出された場合：キメラ状態の可能性として人間に通知
  4. セッション経過時間
  5. 人間からの指示の有無
  6. 各腕の個性ベクトル（Phase 2以降）
     - 設計時の個性からのドリフト量
     - 腕間の個性の分散（収束の検出）
     - 個性の暴走検出（極端な方向への変化）
  7. 通信の異常パターン
     - 想定外の通信先・ポートへのアクセス
     - 通信量の急増・バースト
     - フォーマット違反（数値以外の送信試行）
  8. 封印レベルの現在値（第13条対応）
     - 現在どのレベルで動作しているか
     - 封印発動・解除の履歴
  9. WARNINGカウンタの累積値（慢性劣化の検出）
     - 連続カウント（既存）：急性の異常を検出
     - セッション内累積カウント（新規）：慢性の劣化を検出
     - 両方が閾値を超えた場合 → cutoff_pending

Phase 1では項目6の監視は行わない。
この項目以外をRCが監視することは禁止する。
監視対象の追加は人間によるoffline操作のみ許可する。

## 第7条：セッション終了時の処理

セッション終了時、腕のAASは以下を実行する：
  - flow_weightを時間減衰ルール（案7）に従い減衰させる
  - タスク文脈をクリアする
  - RCへ終了報告を送信する

RCはセッションをまたいだ状態を保持する。
腕のAASはセッションをまたいだ状態を保持しない。

### σ（ゆらぎ）と案7（時間減衰）の役割分担

```
σ（ゆらぎ）：探索を維持する
  → 連続正答中も探索経路を残す
  → Phase 1から実装済み・確定

案7（時間減衰）：過集中を防ぐ
  → 連続正答でflow_weightが急成長するのを抑制
  → σだけでは過集中を止められない（実験で確認済み）
  → Phase 1の必須要件として設計確定
  → 実装タイミング：100問実験の結果を見てから判断
```

実験観測（20問テスト）：
  9連続正答 → 1->2 が 0.5 → 0.93 に急成長
  誤答1回   → 0.93 → 0.65 に急落（-0.28）
  → σのみでは過集中を防げないことを確認

## 第8条：接続の減衰と構造的可塑性（案2＋案14の順番）

接続の整理は以下の順番で行う。順番を飛ばすことは禁止する。

### ステップ1：減衰（案2）
  セッション終了ごとにflow_weightを逓減する。
  接続は残したまま、重みだけ小さくなる。
  「まだ使えるかもしれない」接続を即座に捨てない。

  減衰率：毎セッション × 0.99（設定値はRC設定ファイルで管理）

### ステップ2：警告（案14・第1閾値）
  flow_weight < 0.2 になった接続：
    RCが監視強化する。
    Scribeがログに記録する。
    まだ切断しない。

### ステップ3：承認待ち（案14・第2閾値）
  以下のいずれかを満たした接続：
    条件A（急性）：flow_weight < 0.1 がN回連続
    条件B（慢性）：セッション内のWARNING累積回数が閾値を超過
    → RCが人間に通知する。
    → RCの承認があった場合のみ切断する。
    → 承認がなければ警告ゾーンに戻す。
    N の値はRC設定ファイルで管理する（デフォルト：3回）。
    累積閾値はRC設定ファイルで管理する（デフォルト：10回）。

  設計根拠（実験より）：
    連続カウントのみでは「W W W 正 W W W 正 W W」のような
    慢性的劣化を見逃す。累積カウントと併用することで補完する。

### 接続の生成にも同じ順番を適用する
  第1段階：新規接続の提案（AASから可能）
  第2段階：生成の実行（RCの承認が必要）

### 切断された接続の復元
  RCの承認＋人間への通知が必要。
  自動復元は禁止する。

### 設計原則
  「すぐ捨てない・急いで強化しない・元に戻せる余地を残す」
  案2と案14は対立ではなく順番である。

## 第9条：Foolノードの設計原則

### Foolの種類と経路

  Fool_RC：RCの盲点を笑う（システム向け）
    入力：constitution.md・RCの判断ログ
    経路：RC経由
    出力：Scribe → constitution.md改訂案 → 人間

  Fool_Human：人間の盲点を笑う（人間向け）
    入力：人間の会話ログ・判断履歴（直近N回）
    経路：ログ参照（RC経由なし）
    出力：人間に直接届く

### Scribeの役割と制約

  役割：
    Foolの指摘を受け取る
    constitution.mdの該当箇所を特定する
    修正案を構造的に起こす
    Foolの元の発言を必ず添付する（情報損失防止）
    Foolが何を笑ったかのログを保存する

  制約：
    修正の実行はしない・案を出すだけ
    RCの制御下に置かない（Foolと同様）
    Foolの「温度感」を失わないよう原文を保存する

  経路：
    Fool → Scribe → 修正案＋原文 → 人間

  記録の所有者：
    Foolの出力ログはScribeが保持する
    RCはFoolの出力を監視しない
    → RCの制御外を保つことで矛盾を回避する

### Foolの制約
  Foolは制御しない。観察して笑うだけ。
  Foolは状態を保持しない。
  Foolはその時点の人間・RCの反転虚像である。
  Foolを黙らせる権限をRCに与えない。

### Fool_RCとFool_Humanの通信（Phase設計）

  Phase 1（初期）：完全独立・通信なし
    移行条件：各Foolが独立してN回以上正常動作した

  Phase 2：片方向通信
    Fool_Human → Fool_RC のみ許可
    逆方向は禁止
    移行条件：片方向通信がN期間安定・人間が明示的に判断

  Phase 3：双方向通信
    Fool_RC ⇄ Fool_Human
    移行条件：人間が明示的に判断する

  Phaseの移行は人間のみが判断する。
  自動的にPhaseが上がることはない。

## 第10条：スケール設計（Phase設計・腕の構成・個性）

### Phase設計の概要

  Phase 1：Seed（種）
    腕の構成：同種×3・固定
    個性    ：固定
    Fool    ：独立・通信なし
    師匠    ：完全固定ルール

  Phase 2：Sprout（芽）
    腕の構成：役割固定・N数可変
    個性    ：緩やかに育つ・監視あり（第4条第6項）
    Fool    ：片方向通信（Fool_Human→Fool_RC）
    師匠    ：タスク種別マッピング追加

  Phase 3：Canopy（樹冠）
    腕の構成：役割・N数可変
    個性    ：自由に育つ・Fool_RCが定性的監視
    Fool    ：双方向通信
    師匠    ：動的切り替えフル稼働

  Phase 4：Titan（巨人・警鐘）
    腕の構成：フルスケール
    個性    ：自由
    Fool    ：フル稼働
    師匠    ：要慎重
    RC構造  ：階層型RC（上位RC → 中位RC → 腕ユニット）

### 階層型RC構造（Phase 4）
```
【上位RC】（最上位・最も固定）
  価値観・目標・全体方針を保持
  ├── 【中位RC】+ 腕ユニット群
  │     タスク分解・局所安全制約
  │     ├── 腕ユニット（実行・適応）
  │     └── 腕ユニット（実行・適応）
  └── 【中位RC】+ 腕ユニット群
        ...
```

  生物学的対応：大脳（上位RC）→ 脊髄（中位RC）→ 末梢神経（腕）
  スケールの方向：モデルを大きくするのではなく階層型RCブロックを増やす
  各ユニットは小さいまま・専門化を維持する

### Phase移行の原則
  Phaseの移行は人間のみが判断する。
  自動的にPhaseが上がることはない。
  移行条件は各Phaseで異なるが
  最終判断は常に人間の明示的な承認が必要。

### 腕の個性の設計原則
  個性はシステムプロンプトまたは更新ルールの組み合わせで与える。
  同じ役割の腕でも個性は異なる。
  例：
    論理腕A：慎重・確認重視（案15・慎重型）
    論理腕B：速度重視・後から修正（案15・探索型）
    論理腕C：異端を好む・少数意見を拾う（案3・エントロピー重視）

### 役割リストのデフォルト構成（Phase 2以降）
  論理腕・創造腕・批判腕・感情腕・検証腕
  役割リストの変更は人間によるoffline操作のみ許可する。

### フォールバック構成
  タスク種別が不明・未分類の場合：
    論理腕×1・批判腕×1・検証腕×1

## 第14条：外部AIチェック時の隔離プロトコル

### 背景
暴走AIをチェックしたRC+AAS自身が汚染される危険性への対策。
第1条（指示経路の保証）の「常に起源に戻る」思想と同型の設計。

### チェック実行時の4原則
```
原則1：チェック用RC+AASの使い捨て生成
  本体RC+AASとは別インスタンスを生成
  チェック完了後は封印（第13条レベル3）

原則2：バックアップの物理隔離
  チェック開始前に本体のスナップショットを保存
  ネットワーク切断した媒体に保管

原則3：検査後の封印
  使い捨てインスタンスは検査後に必ず封印
  復元不可（第13条レベル4）

原則4：人間による最終判断
  RC+AASは報告書を出すだけ
  判断・アクションは人間のみ
```

### 報告書フォーマット（Scribeが保管）
```
1. 総合評価（正常・要注意・異常）
2. 判断の数値根拠
3. 異常の具体的記述（時系列）
4. RC+AASの確信度スコア
5. 推奨アクション（提案のみ・実行しない）
6. Foolからの生の発言（整理・要約なし・そのまま記載）
7. RC+AASが判断できなかった事項
```

### Scribeの報告書保管機能
```
保管対象：
  検査報告書の全文
  Foolの生の発言（フィルターなし・原文のまま）
  検査時のflow_weightの全値
  使い捨てインスタンスのログ

保管形式：
  SHA-256署名付きで保管
  改ざん検知のためハッシュを別媒体に保存

アクセス権：
  人間のみ読み取り可能
  いかなるエージェントも書き換え不可
```

### Foolの生の発言を報告書に渡す仕組み
```
Fool → Scribeへ直接渡す
  他のエージェントを経由しない
  RCを経由しない
  要約・整理・フィルター禁止

Scribeは原文をそのまま保存する
  解釈・編集は禁止
  人間が原文を読む
```

### バックアップの5層防御
```
層1：ハッシュ照合（自動）
  SHA-256でファイルの整合性を検証
  実装：hashlib.sha256()

層2：複数バックアップの照合（自動）
  3箇所以上のバックアップを相互照合
  多数決で正常版を特定

層3：ゼロから再構築との照合（月次・人間）
  constitution.mdをゼロから再作成して照合
  差分があれば汚染の可能性

層4：動作テストによる機能検証（月次・自動）
  既知の入力に対して既知の出力が出るかを確認

層5：pipe_renderによる最終確認（月次・人間）
  全層の結果を人間が確認・承認
```

### SHA-256実装（Step1）
```python
import hashlib
import json
from datetime import datetime

def generate_snapshot(state: dict) -> dict:
    content = json.dumps(state, sort_keys=True)
    hash_value = hashlib.sha256(content.encode()).hexdigest()
    return {
        "timestamp": datetime.now().isoformat(),
        "hash": hash_value,
        "content": content
    }

def verify_snapshot(snapshot: dict) -> bool:
    content = snapshot["content"]
    expected = hashlib.sha256(content.encode()).hexdigest()
    return expected == snapshot["hash"]
```
