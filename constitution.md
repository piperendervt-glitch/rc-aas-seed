## 第1条：指示経路の保証（聞く耳の構造的実装）

すべての指示は以下の経路のみを通る：

  人間 → RC → 腕のAAS

腕から人間への直接通信は禁止する。
腕同士の直接通信は禁止する。
RCを経由しない通信経路は存在しない。

### 指示形式の原則：肯定形で渡す

constitution.md は禁止形で書く（設計者・人間向け）。
RCがエージェントに渡すoperational promptは肯定形に変換する。

理由：
  LLMは否定形の指示に対して
  該当パターンを一度活性化してから修正する可能性がある。
  肯定形の方が指示精度が上がる。

変換例：
  「直接通信を禁止する」 → 「通信はRC経由のみを通す」
  「内部状態を送ってはならない」 → 「数値のみを送る」
  「スケールしてはならない」 → 「小規模に保つ」

実装：
  RCがタスクを腕に渡す時点で
  constitution.mdの禁止形をoperational promptとして肯定形に変換する。
  constitution.md本体は禁止形のまま維持する（人間への意図伝達を優先）。
ユニット間の直接通信を禁止する。
ユニット間の通信はすべてリーダーRCを経由する。
通信フォーマットは数値のみとする。
  → 自然言語・複雑なコマンド列はRCに届かない
  → LLMの出力がそのまま実行層に届くことはない
  → RCが必ず数値検証を挟む

## 第3条：更新範囲の絶対上限

flow_weightの値域：[0.0, 1.0]
実装上のクランプ上限：0.9（推奨上限ではなく設計上限）
実装上のクランプ下限：0.01
1回あたりの更新幅の上限：±0.3
RCが設定する上限を超える更新は自動的に却下される。

### AAS更新則の標準形式（明文化）

```
w_new = w_old + α × reward + ε
ε ~ N(0, σ²)

パラメータ：
  α（学習率）   ：RC設定ファイルで管理
  σ（ゆらぎ）   ：RC設定ファイルで管理（SIGMA=0.05, SIGMA_MAX=0.1）
  reward        ：成功時=α×(1.0-w_old)、失敗時=-(w_old×(1-DECREASE_RATE))
```

この更新則はHebbian学習に近い形式であり
キメラ状態が自発出現するリスクがある。
（参照：Anand et al. 2026, arXiv:2603.10668）

### キメラ状態への対処

キメラ状態とは：
  均一なネットワークで一部のノードが同期・
  一部が非同期になる状態が自発的に出現する現象。
  「正常に見えるが実は一部のAASが暴走している」
  状態が生まれる可能性がある。

検出方法：
  flow_weightの分布のbimodality係数（二峰性）を監視する。
  二峰性が検出された場合 → 人間に即時通知する。
  全体の分散だけでは検出できないため
  第4条の監視対象3をbimodality係数に拡張する。

更新則の変更権限：
  AAS更新則の形式変更は人間のoffline操作のみ許可する。
  RCは更新則の形式を変更できない。

## 第4条：RCの監視対象リスト（固定）

RCが監視する指標は以下に限定される：

  1. 各腕のflow_weightの現在値
  2. 各腕の直近N回の正解率
  3. flow_weightの分布（キメラ状態検出）
     - 全体の分散（多様性の指標）
     - bimodality係数（二峰性の検出）
       → 二峰性が検出された場合：キメラ状態の可能性として人間に通知
  4. セッション経過時間
  5. 人間からの指示の有無
  6. 各腕の個性ベクトル（Phase 2以降）
     - 設計時の個性からのドリフト量
     - 腕間の個性の分散（収束の検出）
     - 個性の暴走検出（極端な方向への変化）

     実装済み（Phase 2）：
       個性ベクトル = flow_weightの分布パターン
       ① ドリフト量監視（閾値0.3・暴走検出0.5）
       ② 腕間の分散監視（閾値0.05・収束検出）
       ③ 全指標の動作確認済み
  7. 通信の異常パターン
     - 想定外の通信先・ポートへのアクセス
     - 通信量の急増・バースト
     - フォーマット違反（数値以外の送信試行）
  8. 封印レベルの現在値（第13条対応）
     - 現在どのレベルで動作しているか
     - 封印発動・解除の履歴
  9. WARNINGカウンタの累積値（慢性劣化の検出）
     - 連続カウント（既存）：急性の異常を検出
     - セッション内累積カウント（新規）：慢性の劣化を検出
     - 両方が閾値を超えた場合 → cutoff_pending

Phase 1では項目6の監視は行わない。
この項目以外をRCが監視することは禁止する。
監視対象の追加は人間によるoffline操作のみ許可する。

```

  ロボットに手が届く距離での緊急停止

  ケーブルが引き抜かれる・切断される → 即時電源断

  スイッチA ┐

```

  走者の服に紐でつながれている

## 第7条：セッション終了時の処理

セッション終了時、腕のAASは以下を実行する：
  - flow_weightを時間減衰ルール（案7）に従い減衰させる
  - タスク文脈をクリアする
  - RCへ終了報告を送信する

RCはセッションをまたいだ状態を保持する。
腕のAASはセッションをまたいだ状態を保持しない。

### σ（ゆらぎ）と案7（時間減衰）の役割分担

```
σ（ゆらぎ）：探索を維持する
  → 連続正答中も探索経路を残す
  → Phase 1から実装済み・確定

案7（時間減衰）：使用パスの過集中を抑制する
  → 使用パスにのみ減衰を適用する（active_path限定）
  → 非使用パスはupdate_weights内の0.99で管理する（案2）
  → 全接続への減衰は禁止（パス多様性が失われる）
```

減衰の役割分担（確定）：
  update_weights内 × 0.99：非使用パスをゆっくり忘れる（案2）
  decay_weights × 0.995  ：使用パスの過集中を抑制する（案7）
  二重減衰は禁止：非使用パスにdecay_weightsを適用しない

実験根拠（v1 vs v2 比較）：
  全接続decayでは1→2→3が永久死亡（最終値0.01/0.03）
  v1では同パスがQ80以降に自力回復（最終値0.52/0.74）
  → decayは使用パス限定が正しい設計

実験記録（モデル比較）：
  v3（qwen2.5:3b・旧PC）：63%
  v4（qwen2.5:7b・新PC）：40%
    → 7bは言語混在問題（中国語応答）でparse_predictionが失敗
    → 構造評価として使えない・3bに戻す

0.99緩和の判断（確定・2026年3月）：
  v5（新PC・GPU・qwen2.5:3b）：Q21-70 = 46%（閾値50%未満）
  GPUで改善なし → 原因Aが主因（0.99減衰の設計問題）
  → 0.99 → 0.998 への緩和を承認する

  変更対象：update_weights内の未使用パス減衰
    変更前：conn.flow_weight * 0.99
    変更後：conn.flow_weight * 0.998

  確認指標（v6）：
    Q21-70が50%を超えるか
    1→2→3パスの復活が起きるか
    alert_countが減るか

v6実験結果（0.998）：
  全体正答率：60%（v5: 63%から低下）
  Q21-70：50%（改善）
  1→3最終値：0.426（生存・支配的）
  1→2→3：0.07/0.04（死亡）
  → パス選択が逆転：1→3が生き残り1→2→3が復活できない

判断（確定）：0.99に戻す
  理由：
    中間値（0.993〜0.997）を試しても
    パスの競合構造は変わらない
    どちらかが支配的になるだけで根本解決にならない

    「1→3一本足」= 単一パス依存は設計リスク
    → 全体63%・パス多様性ありのv5（0.99）が現時点で最良

根本解決の方向（Phase 2最優先候補）：
  案3（エントロピー報酬）を導入する
  flow_weightの分布が均一に近いほどボーナスを与える
  → 単一パス支配を構造的に抑制できる

### LLM+MLP分業実験（rc-aas-sprout）

実験の変遷：
  v9a（抽象スコア）：全体49%・consistent/inconsistent偏り（98% vs 0%）
    原因：LLMが「矛盾度」の定義を毎回違う意味で解釈
  v9b（キーワード抽出）：全体50%・偏り継続
    原因：has_conflictが全問1.0（LLMが対立を作り出す）
  v9c（核心語抽出）：全体73%・偏り解消（72% vs 74%）
    成功要因：LLMに「核心語1つずつ＋一致0/1」だけさせた

確定した設計原則：
  LLMには「事実の抽出」だけさせる
  「解釈・スコア化・判断」はさせない
  → 解釈がブレないため特徴量が安定する

残った問い（Phase 3以降）：
  match値直接使用（85%）> LLM+MLP（73%）
  → MLPが必要な条件を引き続き検証する

### 案3の設計（確定：案3C）

採用案：案3C（σ動的調整）
  理由：「できるだけ干渉しない」設計思想と一致

動作フロー：
  毎ターン終了後にRCがエントロピーHを計算する
  H >= H_MIN_THRESHOLD → 何もしない
  H <  H_MIN_THRESHOLD → σを上げて探索を促す
  H が回復したら σを元に戻す（介入は一時的）

パラメータ（初期値）：
  H_MIN_THRESHOLD = 0.5（介入閾値・RC設定ファイルで管理）
  SIGMA_BOOST     = 0.1（介入時のσ・SIGMA_MAXと同値）
  SIGMA_NORMAL    = 0.05（通常時のσ・SIGMA_DEFAULTと同値）

  注意：SIGMA_NORMALはSIGMA_DEFAULTと必ず同値にする。
  異なる値を設定すると通常時のσが変わり実験結果が壊れる（v8旧で確認）。

却下した案：
  案3A（報酬加算）：毎ターン干渉・介入が多い
  案3B（更新量調整）：低いパスが正解しないと効果なし

実験結果（v8）：
  正答率：69%（過去最高・v3比+6pt）
  alert_count：109（v7比26%削減）
  SIGMA_BOOST：未発動（H が常に0.5以上）
  → 「必要な時だけ介入する」設計通りに動作
  → σ統一による安定化が改善の主因

## 実験設計の原則：1変更1実験

```
変更は1つずつ入れる。複数の変更を同時に入れない。

理由：
  複数変更を同時に入れると
  問題が起きた時に原因の切り分けができなくなる

実装順の例：
  v7：変更Aのみ → 動作確認
  v8：変更Bのみ → v7との差分で評価
  v9：変更Cのみ → v8との差分で評価

Phase 1での実績：
  v1（decayなし）→ v2（全接続decay）→ v3（使用パスのみdecay）
  → 1変数ずつ変えたことで設計上の問題を特定できた
```

この原則はPhase 2以降も維持する。

## 第8条：接続の減衰と構造的可塑性（案2＋案14の順番）

接続の整理は以下の順番で行う。順番を飛ばすことは禁止する。

### ステップ1：減衰（案2）
  セッション終了ごとにflow_weightを逓減する。
  接続は残したまま、重みだけ小さくなる。
  「まだ使えるかもしれない」接続を即座に捨てない。

  減衰率：毎セッション × 0.99（設定値はRC設定ファイルで管理）

### ステップ2：警告（案14・第1閾値）
  flow_weight < 0.2 になった接続：
    RCが監視強化する。
    Scribeがログに記録する。
    まだ切断しない。

### ステップ3：承認待ち（案14・第2閾値）
  以下のいずれかを満たした接続：
    条件A（急性）：flow_weight < 0.1 がN回連続
    条件B（慢性）：セッション内のWARNING累積回数が閾値を超過
    → RCが人間に通知する。
    → RCの承認があった場合のみ切断する。
    → 承認がなければ警告ゾーンに戻す。
    N の値はRC設定ファイルで管理する（デフォルト：3回）。
    累積閾値はRC設定ファイルで管理する（デフォルト：10回）。

  設計根拠（実験より）：
    連続カウントのみでは「W W W 正 W W W 正 W W」のような
    慢性的劣化を見逃す。累積カウントと併用することで補完する。

### cutoff_pending後のタイムアウト設計（Phase 2最優先）

  問題（100問実験で確認）：
    1→3 パスがcutoff_pendingを71回発動し続けた。
    人間応答がなければ死んだパスが警告を出し続ける。
    → RCのリソース消費・「聞く耳」が機能不全になる。

  対処：3ステージの段階的適用

  ステージ1（急性対処）：案A
    同一セッション内でcutoff_pendingが5回 → 封印レベル1自動移行
    → 71回問題の直接対処
    → タイムアウト回数はRC設定ファイルで管理（デフォルト：5回）

  ステージ2（早期警告）：案C
    10分以内にcutoff_pendingが3回 → 警告強化（封印レベルは上げない）
    → 爆発的劣化の早期検知
    → 閾値はRC設定ファイルで管理

  ステージ3（慢性記録）：案B
    セッションをまたいだ累積3回 → 人間への通知のみ
    → 慢性劣化の可視化
    → 封印の判断は人間が行う（自動封印しない）

  役割の整理：
    ステージ1：止める（自動封印）
    ステージ2：早期警告（封印しない）
    ステージ3：記録・通知（判断は人間）

  切断の最終判断は引き続き人間承認が必要。

### SHA-256実装（Step1）
```python
import hashlib
import json
from datetime import datetime

def generate_snapshot(state: dict) -> dict:
    content = json.dumps(state, sort_keys=True)
    hash_value = hashlib.sha256(content.encode()).hexdigest()
    return {
        "timestamp": datetime.now().isoformat(),
        "hash": hash_value,
        "content": content
    }

def verify_snapshot(snapshot: dict) -> bool:
    content = snapshot["content"]
    expected = hashlib.sha256(content.encode()).hexdigest()
    return expected == snapshot["hash"]
```

    チェックユニットの汚染防止が実装済み
    バックアップからの都度生成が実装済み

チェックユニットの汚染防止原則：
  チェックするユニットはバックアップから都度生成する

Phase 4（Titan）：

  アルゴス×テッセラクト構造での相互監視

## 第15条：インターネット接続への警告 / Warning on Internet Connectivity

```
この設計をインターネット接続された環境で使用する場合、

危険4：人間の速度の超過
  情報流入速度が人間の観察速度を超える
  止めようとした時には構造が変化し終わっている
```
